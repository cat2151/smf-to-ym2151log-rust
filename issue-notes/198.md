# issue 波形ビュアとWAVエクスポートが、「3.変換結果」を反映していない #198
[issues #198](https://github.com/cat2151/smf-to-ym2151log-rust/issues/198)

## Root Cause Analysis

The waveform viewer was generating a "meaningless" waveform (独自に生成した波形)
that looked the same regardless of whether PopNoiseEnvelope / AttackContinuationFix
was enabled or disabled in the conversion result.  Two compounding bugs caused this.

### Bug 1 — Waveform viewer used a self-made sine simulation instead of real audio

The viewer contained `waveform-simulator.ts` and `envelope-generator.ts` which
generated samples entirely in JavaScript as `envelope × sin(phase)`.  This bore no
relationship to the audio actually produced by `web-ym2151`: it ignored FM modulation,
operator connections, LFO, and the exact ADSR curves of the real YM2151 WASM engine.
Any visual difference between "pop-noise on" and "pop-noise off" was at best
coincidental.

**Fix:** `waveform-simulator.ts` and `envelope-generator.ts` are deleted.
`waveform-viewer.ts` now calls `generateAudioFromJson(jsonText)` from the `web-ym2151`
audio module, which returns real PCM samples (`AudioData.left`) produced by the same
WASM synthesizer used for audio playback.  Note-boundary markers (key-on timestamps for
the selected channel) are extracted directly from the JSON event list.

### Bug 2 — Default `AttackContinuationFix.ReleaseRate` had wrong encoding (`pop-noise-demo.ts`)

The default attachment used `"ReleaseRate": 240` (= 0xF0 hex).  This value is
written directly as the full D1L/RR register byte:

| Byte   | D1L (bits 7:4) | RR (bits 3:0) |
|--------|----------------|---------------|
| `0xF0` | 15 (max)       | **0 (no release!)** |

With RR = 0, the envelope amplitude does **not move** during the brief release window.
The feature was effectively disabled.

The correct byte for D1L=15, RR=15 (maximum release rate) is `0xFF = 255`.

**Fix:** `"ReleaseRate": 240` → `"ReleaseRate": 255`.

### Background — Why the old simulation never showed a difference

The old `EnvelopeGenerator` defaults (D1L = 15, RR = 7) gave a ~1.48 million-second
release — amplitude stayed near 1.0 forever.  Any mitigation feature that failed to
change the actual RR nibble had zero visible effect.

## Visible Result After Fix

The waveform now shows the real YM2151 output.  Pop-noise appears as a sudden
amplitude discontinuity at note boundaries; `AttackContinuationFix` smooths this by
forcing amplitude to ~0 (in ~0.36 ms with RR=15) before the next key-on.

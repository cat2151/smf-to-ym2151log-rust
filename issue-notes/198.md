# issue 波形ビュアとWAVエクスポートが、「3.変換結果」を反映していない #198
[issues #198](https://github.com/cat2151/smf-to-ym2151log-rust/issues/198)

## Root Cause Analysis

The waveform viewer was generating a "meaningless" waveform (独自に生成した波形)
that looked the same regardless of whether PopNoiseEnvelope / AttackContinuationFix
was enabled or disabled in the conversion result.  Two compounding bugs caused this.

### Bug 1 — Oscillator phase never reset on key-on (`waveform-simulator.ts`)

Pop noise on the YM2151 occurs as an audio *click* when a new note's key-on fires
while the previous note's envelope amplitude is still non-zero.  The click arises
from the sudden phase discontinuity: the previous carrier sine was at angle φ
(producing `amplitude × sin(φ)`), and the new note's sine starts fresh from 0
(producing `amplitude × sin(0) = 0`).

The simulator never reset `oscPhase` on key-on, so the carrier sine wave was
*continuous* across note boundaries — no phase jump, no visible click, regardless of
the amplitude.  The waveform therefore looked identical whether or not pop-noise
mitigation was active, making it appear "independent" of the conversion result.

**Fix:** `oscPhase = 0` is now added immediately after `gen.keyOn()`.

### Bug 2 — Default `AttackContinuationFix.ReleaseRate` had wrong encoding (`pop-noise-demo.ts`)

The default attachment used `"ReleaseRate": 240` (= 0xF0 hex).  This value is
written directly as the full D1L/RR register byte:

| Byte   | D1L (bits 7:4) | RR (bits 3:0) |
|--------|----------------|---------------|
| `0xF0` | 15 (max)       | **0 (no release!)** |

With RR = 0, `ampStepPerSample(0) = 0`, so the envelope amplitude does **not move**
during the brief release window.  The feature was effectively disabled.

The intended behaviour is a *fast* release so that amplitude drops to ~0 before the
next note-on.  The correct byte for D1L=15, RR=15 (maximum release rate) is
`0xFF = 255`.  With RR = 15, `ampStepPerSample(60) ≈ 0.052` per sample, and
amplitude reaches 0 in about 20 samples (~0.36 ms) — well within the 1 ms offset
window.

**Fix:** `"ReleaseRate": 240` → `"ReleaseRate": 255`.

### Background — Why the envelope never decays otherwise

The YM2151 initialisation (`initialize_channel_events`) sets D1L = 15 and D2R = 5
for all operators.  D1L = 15 means "sustain at 100 % of attack peak", so decay-1
completes instantly and decay-2 progresses at an effectively infinite rate (~388 billion
seconds per traversal at effective rate 10).  The release rate RR = 7 gives ~1.48
million seconds per traversal — so amplitude stays
near 1.0 forever.  Any mitigation feature that fails to change the actual RR field has
zero visible effect on the simulated waveform.

## Visible Result After Fix

| Condition | Waveform |
|-----------|----------|
| No mitigation | Amplitude ≈ 1.0 at note-on; oscPhase jumps to 0 → visible click |
| AttackContinuationFix ON (ReleaseRate=255) | Amplitude drops to 0 in ~0.36 ms before note-on; oscPhase jumps to 0 at amplitude≈0 → smooth |

The waveform now clearly reflects whether the conversion result contains effective
pop-noise mitigation events.
